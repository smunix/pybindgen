  2  """ 
  3   
  4  Introduction 
  5  ============ 
  6   
  7    PyBindGen is a Python module that lets you generate python language bindings for C or C++ code. 
  8   
  9  How to use it 
 10  ============= 
 11   
 12   There are multiple ways to use PyBindGen, and they are described in the following subsections. 
 13   
 14   Basic Interface 
 15   --------------- 
 16   
 17    PyBindGen does not have any command-line interface.  The author 
 18    needs to write a small python script that imports the pybindgen 
 19    module and uses it to generate the bindings.  For instance:: 
 20   
 21      #! /usr/bin/env python 
 22   
 23      import sys 
 24   
 25      import pybindgen 
 26      from pybindgen import (ReturnValue, Parameter, Module, Function, FileCodeSink) 
 27   
 28      def my_module_gen(out_file): 
 29          pybindgen.write_preamble(FileCodeSink(out_file)) 
 30   
 31          mod = Module('a') 
 32          mod.add_include('"a.h"') 
 33   
 34          mod.add_function('ADoA', None, []) 
 35          mod.add_function('ADoB', None, [Parameter.new('uint32_t', 'b')]) 
 36          mod.add_function('ADoC', ReturnValue.new('uint32_t'), []) 
 37   
 38          mod.generate(FileCodeSink(out_file) ) 
 39   
 40      if __name__ == '__main__': 
 41          my_module_gen(sys.stdout) 
 42   
 43   
 44    Running the above python script will generate to stdout the source code for a C library with the following interface:: 
 45   
 46      #include <stdint.h> 
 47   
 48      void ADoA (void); 
 49      void ADoB (uint32_t b); 
 50      uint32_t ADoC (void); 
 51   
 52    The most important classes to retain from pybindgen are: 
 53   
 54      - L{CodeSink}, which is an object that pybindgen uses to receive 
 55        the generated code.  Typically we want to use a subclass of it 
 56        called L{FileCodeSink}, which receives generated code and writes 
 57        it to a file-like object; 
 58   
 59      - L{Module}, which represents a Python extension module.  The 
 60        progammer can L{add functions<Module.add_function>}, 
 61        L{classes<Module.add_class>}, L{enums<Module.add_enum>}, or even 
 62        L{other modules<Module.add_cpp_namespace>} to a module object.  The 
 63        module object is the main front-end for L{code 
 64        generation<Module.generate>}, and it "owns" every other object. 
 65   
 66      - Then there are objects to represent L{functions<Function>}, 
 67        L{C++ classes<CppClass>}, L{enumerations<Enum>}, etc.  These 
 68        additionally make use of objects that represent L{return 
 69        values<ReturnValue>} and L{parameters<Parameter>}. 
 70   
 71   Basic Interface with Error Handling 
 72   ----------------------------------- 
 73   
 74    It is also possible to declare a error handler.  The error handler 
 75    will be invoked for API definitions that cannot be wrapped for some 
 76    reason:: 
 77   
 78      #! /usr/bin/env python 
 79   
 80      import sys 
 81   
 82      import pybindgen 
 83      from pybindgen import Module, FileCodeSink, retval, param 
 84   
 85      import pybindgen.settings 
 86      import warnings 
 87   
 88      class ErrorHandler(pybindgen.settings.ErrorHandler): 
 89          def handle_error(self, wrapper, exception, traceback_): 
 90              warnings.warn("exception %r in wrapper %s" % (exception, wrapper)) 
 91              return True 
 92      pybindgen.settings.error_handler = ErrorHandler() 
 93   
 94   
 95      def my_module_gen(out_file): 
 96          pybindgen.write_preamble(FileCodeSink(out_file)) 
 97   
 98          mod = Module('a') 
 99          mod.add_include('"a.h"') 
100   
101          mod.add_function('ADoA', None, []) 
102          mod.add_function('ADoB', None, [param('uint32_t', 'b')]) 
103          mod.add_function('ADoC', retval('uint32_t'), []) 
104   
105          mod.generate(FileCodeSink(out_file) ) 
106   
107      if __name__ == '__main__': 
108          my_module_gen(sys.stdout) 
109   
110    In this example, we register a error handler that allows PyBindGen 
111    to simply ignore API definitions with errors, and not wrap them, but 
112    move on. 
113   
114    The difference between is Parameter.new(...) and param(...), as well 
115    as between ReturnValue.new(...) and retval(...) is to be noted here. 
116    The main difference is not that param(...) and retval(...) are 
117    shorter, it is that they allow delayed error handling.  For example, 
118    when you put Parameter.new("type that does not exist", "foo") in 
119    your python script, a TypeLookupError exception is raised and it is 
120    not possible for the error handler to catch it.  However, param(...) 
121    does not try to lookup the type handler immediately and instead lets 
122    Module.add_function() do that in a way that the error handler can be 
123    invoked and the function is simply not wrapped if the error handler 
124    says so. 
125   
126   Header file scanning with (py)gccxml 
127   ------------------------------------ 
128   
129    If you have gccxml and pygccxml installed, PyBindGen can use them to 
130    scan the API definitions directly from the header files:: 
131   
132      #! /usr/bin/env python 
133   
134      import sys 
135   
136      import pybindgen 
137      from pybindgen import FileCodeSink 
138      from pybindgen.gccxmlparser import ModuleParser 
139   
140      def my_module_gen(): 
141          module_parser = ModuleParser('a1', '::') 
142          module = module_parser.parse([sys.argv[1]]) 
143          module.add_include('"a.h"') 
144   
145          pybindgen.write_preamble(FileCodeSink(sys.stdout)) 
146          module.generate(FileCodeSink(sys.stdout)) 
147   
148      if __name__ == '__main__': 
149          my_module_gen() 
150     
151    The above script will generate the bindings for the module directly. 
152    It expects the input header file, a.h, as first command line 
153    argument. 
154   
155   Header file scanning with (py)gccxml: python intermediate file 
156   -------------------------------------------------------------- 
157   
158    The final code generation flow supported by PyBindGen is a hybrid of 
159    the previous ones.  One script scans C/C++ header files, but instead 
160    of generating C/C++ binding code directly it instead generates a 
161    PyBindGen based Python script:: 
162   
163      #! /usr/bin/env python 
164   
165      import sys 
166   
167      from pybindgen import FileCodeSink 
168      from pybindgen.gccxmlparser import ModuleParser 
169   
170      def my_module_gen(): 
171          module_parser = ModuleParser('a2', '::') 
172          module_parser.parse([sys.argv[1]], includes=['"a.h"'], pygen_sink=FileCodeSink(sys.stdout)) 
173   
174      if __name__ == '__main__': 
175          my_module_gen() 
176    
177    The above script produces a Python program on stdout.  Running the 
178    generated Python program will, in turn, generate the C++ code 
179    binding our interface.
